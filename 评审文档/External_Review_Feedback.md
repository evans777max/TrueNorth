# TrueNorth Skill-first MVP v1 · 外部 Review 反馈

- 评审视角：外部 reviewer / 潜在用户 / 产品评审者
- 评审对象：`TrueNorth.html` + `评审文档/`（开发文档 / Review_Report / P0_P1_P2_Issues / Browser_Walkthrough_Report）
- 评审性质：方向是否成立、Skill-first 是否有价值、用户是否看得懂会用、下一版收敛什么。**只做判断与建议，不改文件。**
- 评审日期：2026-06-19

---

## 1. 总体判断

- **是否成立：成立。** 它确实不是收藏夹/工具黄页——「AI 信号 → 关键洞察 → 实战案例 → 方案/Skill → 跑通」这条主线有结构、有溯源、有"为什么值得用"（whyTrueNorth），方向清楚。
- **最大亮点：Skill-first + whyTrueNorth + sourceRefs/extractedFrom 三件套。** 这是它和普通 AI 站最本质的区别：每个能力单元都告诉你"能干什么、为什么选它、出处在哪"。
- **最大风险：核心资产（Skill 库）的入口被埋，且"跑通"承诺还差一口气。** Skill 库藏在「方案库」的视图切换里，不是一等入口；同时 4/10 条 Skill 没有可复制 prompt、部分 install 写法模糊，"看懂→跑通"在这几条上断了。
- **是否适合外部 review：适合**（朋友、AI 工具用户、工程师都可以）。
- **是否适合公开发布：需补充后可以**（移动端横向溢出未实测 + 部分 Skill 不可即用 + 内容量仍为 MVP 级）。
- **是否建议继续扩容：暂不。** 先收敛主路径与补全现有 10 条 Skill，再谈扩容。

## 2. 产品定位评估

**清楚的地方**
- Hero 主标题「把 AI 信息，变成你能直接用的工作打法」一句话说清了价值主张。
- QuickStart「不知道先看什么？从这里开始」+ 三个任务式入口，对新用户友好。
- Route 用 01→06 的路线把"看懂→跑通"可视化，叙事完整。

**不清楚的地方**
- **首屏没点名 Skill-first。** 产品的核心身份是"Skill-first 可执行能力层"，但 Hero 文案与三个 CTA（看 AI 信号 / 找可复用方案 / 从 0 读懂 AI）里都没直接出现"Skill"。用户要到方案库内部切视图才发现 Skill 库——核心卖点在首屏是隐身的。
- Hero 副标题「把分散信息拆解成可溯源、可复用、可执行的工作参考」偏长、偏抽象，"可执行"这个最关键的词被淹没。

**一句话定位建议**：对外统一成「**把 AI 信息变成可复制的 Skill 和能跑通的方案**」——把"Skill / 能跑通"提到最显眼处。

**首屏文案是否需要调整**：建议（P1）。把第三个 CTA 或新增一个 CTA 直接写成「找可用 Skill →」，让 Skill-first 在首屏可见。

## 3. 用户路径评估

| 用户类型 | 目标 | 当前路径 | 是否顺畅 | 卡点 | 建议 |
| --- | --- | --- | --- | --- | --- |
| A 跟上变化 | 看最近 AI 新变化 | Hero「看本周 AI 信号」/ QuickStart 📡 → AI 信号 → 信号解读 | 顺 | 信号标注"非实时/精选"要够明显，避免误期待 | 维持 |
| B 看别人怎么用 | 找真实案例 | QuickStart 🧩 → 实战案例 → 案例解读 → 相关工具/Skill | 顺 | 无明显卡点 | 维持 |
| C 找工具/方案 | 找可复用方案 | Hero「找可复用方案」→ 方案库（项目视图）→ 详情 → 跑通路径 | 较顺 | 33 个方案靠筛选/搜索，新手可能不知道先点哪个 | 加一行"新手从这 3 个开始"引导 |
| D 找可用 Skill | 直接找 Skill | 方案库 → **手动切到「Skill 库」视图** → Skill 详情 | **不顺（关键卡点）** | Skill 库不是一等入口，藏在方案库的视图 tab 里；与"Skill-first"定位矛盾 | **P1：Skill 库前置为首屏/导航一级入口** |
| E 学概念 | 系统学 AI 概念 | Hero「从 0 读懂 AI」/ Route 05 → AI 词典 → 7 站/故事/卡片 | 顺 | 词典体量大，但有 7 站分层，可接受 | 维持 |

**当前路径是否顺**：A/B/E 顺，C 基本顺，**D 不顺**。
**卡点在哪里**：最核心的 Skill-first 用户（D）反而路径最绕——要先进方案库、再发现并点击视图切换，才能到达 Skill 库。
**最该优化哪个入口**：**Skill 库入口**。这是定位与导航不一致的唯一硬伤。

**关于 QuickStart 与 Route**：两者都有效，但**功能重叠**——QuickStart 三卡（变化/案例/工具）与 Route 六节（信号/洞察/案例/方案/词典/来源）指向高度重合，首页等于连续两层"去哪里"导航。建议差异化：QuickStart = 任务入口（我想干什么），Route = 路线说明（整体怎么走一遍），不要让用户连读两遍相似的入口。

## 4. Skill 库评估

| Skill | 价值判断 | 问题 | 建议 |
| --- | --- | --- | --- |
| sk-brainstorming 先问再做 | 高（受众最广、第一步刚需） | 无 | 当样板 |
| sk-debug 系统化排查 bug | 高（开发者高频、prompt 完整） | 无 | 当样板 |
| sk-prompt-opt 提示词优化器 | 高（通用底座、零门槛、prompt 完整） | install 为空 | 标注"直接用 prompt，无需安装" |
| sk-compress 精准压缩 | 高（信息过载刚需） | install「参考仓库 README」模糊 | 给具体安装命令 |
| sk-tacit 隐性知识挖掘 | 中（团队沉淀价值高，prompt 完整） | 偏专业向 | 维持 |
| sk-teach 让 AI 当老师 | 中（卖点好） | prompt 仅「/teach …」，依赖已装 skill | 补一句最小可用示例 |
| sk-mag-ppt 电子杂志风 PPT | 中高（PPT 高频） | **prompt 为空**，仅靠 install | 补可复制 prompt 或标"安装即用" |
| sk-impeccable 防 AI 辣眼睛设计 | 中（痛点真实） | **prompt 为空 + install「参考 GitHub 仓库…安装」模糊** | 补具体安装 + 用法 |
| sk-ost 机会-方案树 | 中（PM 向、偏窄） | **prompt 为空**、受众窄 | 维持或标注适用人群 |
| sk-prioritize 假设排优先级 | 中（PM 向、偏窄） | **prompt 为空**、受众窄 | 维持或标注适用人群 |

- **最强 3 条**：`sk-brainstorming`、`sk-debug`、`sk-prompt-opt`——受众广、prompt 完整可直接复制、whyTrueNorth 有说服力，最能体现"看懂→跑通"。
- **最弱 3 条**：`sk-impeccable`（空 prompt + 模糊 install）、`sk-ost`、`sk-prioritize`（PM 向偏窄 + 空 prompt）——更像"指针/书签"而非"可即用 Skill"。
- **最有用的字段**：`whyTrueNorth`（这是和链接列表的根本区别，回答"为什么用它"）、`prompt`（有就能直接复制跑）、`sourceRefs.extractedFrom`（可追溯、增强可信度）。「看使用方式」确实比「查看详情」更准——它承诺的是"怎么用"，符合 Skill-first。
- **看起来冗余的字段**：`input/output` 偏样板（如 input「一句粗糙的需求」/ output「对齐后的需求」信息量低）；`relatedGlossary` 仅 3/10 有值，目前偏稀、价值有限；部分 `tags` 只有单个词。
- **下一版最该优化**：① **补齐 4 条空 prompt**（或在卡片明确标"安装即用/无需 prompt"，让用户不困惑）；② **统一 install 写法**，消除"参考仓库 README""参考 GitHub 仓库…安装"这类不可执行的描述。这两件直接决定"跑通"承诺是否兑现。

**Skill 是否能作为"看懂→跑通"的桥梁**：能——但目前是"6 条桥已通、4 条桥还差最后一块板"。

## 5. 内容结构评估

- **信息架构评分：6.5 / 10。** 模块齐全、关系层完整，但**模块偏多 + 首页三层导航 + 模块式（而非任务式）路线**，新用户存在轻度"先看哪"的负担。
- **最重要的 3 个模块**：① Skill 库（核心资产）② 方案库 / 跑通路径（落地）③ 实战案例（说服力）。
- **可以后置/弱化的模块**：来源图谱（信任/溯源价值高，但属"查证"低频动作，可后置）；AI 词典体量大，可作为"支撑层"而非主路径节点。
- **任务入口 vs 模块入口**：建议**强化任务入口**。QuickStart 已是任务式（我想看变化/我想找工具），方向对；Route 是模块式，和 QuickStart 重叠。下一版建议首页收敛：保留一套任务入口 + 一处轻量"完整路线"说明即可。
- **是否建议下一版做首页收敛**：建议（P1，纯文案/版块取舍，不重构）。

## 6. 跑通路径评估

- **最实用**：可部署项目里的「跑通路径 + 准备材料 + 适合谁/不适合谁 + 风险说明 + Codex/Kiro/Claude prompt tab」——对开发者很实在，且诚实标注了适用人群与风险，这点很好。在线工具类（如 Zread「一键读懂 GitHub 项目」）属最轻、最易跑通的一类，体验好。
- **看起来过重**：对**非开发者**而言，带多 tab prompt、安装步骤、准备材料的跑通路径偏重，容易被劝退。
- **prompt 是否有价值**：有（尤其 Skill 里完整的那几条，可直接复制）。
- **下一版如何让"跑通"更轻**：① 每条路径给一个"**最小可行操作**"（一句话/一条命令/一个在线链接先跑起来），把完整步骤折叠为进阶；② 用 `difficulty` 显式区分新手/进阶路径（数据已有 difficulty 字段，前端可强化呈现）。**不需要新增内容**，只是呈现层的轻重分层。

## 7. 文档与评审材料评估

- **文档清晰度评分：8.5 / 10。**
  - 开发文档：项目结构、数据模型、Skill-first、**人工精选 vs 后续脚本辅助**（第 7 节区分清楚）、拆文件阈值、reviewer 检查清单都讲清了，外部工程师能快速上手。
  - Review_Report：结论与代码实际一致，可信。
  - P0_P1_P2_Issues：分级合理，无未解决 P0 写得明确。
  - Browser_Walkthrough_Report：**诚实**——明确写了"本环境非真实图形浏览器、用 jsdom 验证、移动端横向溢出需人工目检"，不谎称已通过。这种坦白反而提升了整套材料的可信度。
- **最需要补充的地方**：文档/About 称"可执行能力层"，但当前 4/10 Skill 不可即用——建议在开发文档加一句**坦白现状**（"10 条中 6 条含可复制 prompt，4 条为安装即用/待补"），消除"过度承诺"的观感。
- **是否适合发给外部工程师 / AI reviewer**：适合，材料齐全且诚实。

## 10. 下一步建议（三阶段）

- **阶段 A · 继续冻结 + 小范围试用（现在）**：把 `TrueNorth_MVP_v1_Final_Review` 发给 3-5 人（含 AI 工具用户 + 至少 1 个手机用户）。只收集三个问题的答案：①一眼能看懂这是什么吗？②能找到 Skill 库吗？③能照着跑通一条 Skill/方案吗？
- **阶段 B · v1.1 小修（试用反馈后）**：① Skill 库入口前置；② 补齐 4 条空 prompt / 统一 install；③ 首页收敛 QuickStart×Route；④ 真实手机目检横向溢出。均为入口、文案、字段级小修，不新增模块、不重构。
- **阶段 C · 扩容前准备（更后）**：Skill 超 30 再拆数据文件；试用反馈稳定后再评估搜索 UI / 来源反向统计；内容扩容放最后。

## 最终判断

- **当前版本是否成立**：成立（方向、模型、材料都立得住）。
- **是否建议继续推进**：建议。
- **是否建议马上公开发布**：不建议——先 v1.1 小修。
- **是否建议先小范围试用**：强烈建议（3-5 人，含手机用户与 AI 工具用户）。
- **下一版最重要的一件事**：**把 Skill 库变成"看得见、点得到、跑得通"的一等入口**——入口前置 + 补全 4 条空 prompt / 模糊 install。

> 一句话结论：**TrueNorth Skill-first MVP v1 值得继续推进——方向成立、Skill-first 有真价值、材料诚实可信；下一步不该堆内容，而该把"最核心的 Skill 库"从"埋在方案库里、四条跑不通"收敛成"首屏可达、条条能跑"。**
